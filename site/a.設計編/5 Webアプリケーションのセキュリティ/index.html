    Webアプリケーションのセキュリティ

    筆者: 海老原 賢次（ebi311@gmail.com）
    作成日: 2023-09-26 更新履歴:

    • 2023-09-26: 新規作成

    Webアプリケーションに限らず、システム・ソフトウェアのセキュリティは、非常に重要な要素です。

    とくに、Webアプリケーションは不特定多数のユーザーがアクセスする事が多く、意図しないアクセスによって脆弱性を疲れることも多いです。

    この章では、Webアプリケーションのセキュリティについて、基本的な考え方を学びます。

    情報

    IPAでは、下記のサイトで情報を提供しています。開発者は全員、こちらを一読してください。

    安全なウェブサイトの作り方 | 情報セキュリティ | IPA 独立行政法人 情報処理推進機構

    社内システムだからセキュリティは甘くてもいい？

    そんな事はありません。XSSやCSRF 攻撃は、ブラウザを介して簡単に外部から発生させることができます。

    また、トロイの木馬などのマルウェアによって、社内ネットワークに侵入された際、セキュリティが甘いと、 例えば一般ユーザーでもDBにアクセスできたり、重要な機能のAPIに認証をかけてなかったりするなどの場合、被害が大きくなります。

    社内だからこそ重要な情報がたくさんあるわけですから、社内システムに対するセキュリティに関する意識を高めてください。

    参考：社内システムのセキュリティとアクセス制御の常識：企業システムの常識をJBossで身につける（9）（1/4 ページ） - ＠IT

    〇〇インジェクション

    SQLインジェクションに代表される、インジェクション攻撃は、非常にメジャーな攻撃手法です。

    インジェクション攻撃はSQL以外にもありえます。いかに一例を示します。

    • HTTPヘッダインジェクション
    • OSコマンドインジェクション
    • URLパス・パラメータインジェクション
    • LDAPインジェクション

    被害としては、下記が挙げられます。

    • DBへ攻撃者が用意したSQLを実行される。データの削除や改ざん、最悪DBごと削除など。
    • サーバー上でOSコマンドを実行される。サーバーの乗っ取りなど。
    • XSSやCSRF（それぞれ後述）に利用される。
    • 攻撃者が用意したページへの勝手な遷移

    SQLインジェクションという言葉から、SQLを使っていないから大丈夫、と考えるのは危険です。

    原因

    これらの脆弱性に共通して言えるのは、入力された値に対して正しいバリデーション、エスケープ処理を行っていない、クエリなどライブラリ等を使わずに文字列結合しているとが原因です。

    攻撃ではなくても、入力された値をそのまま使うことは、バグの原因にもなります。

    これらをサニタイズと呼ぶこともありますが、そもそも実装として間違いなので、純粋なバグと言っても良いでしょう。

    高木浩光＠自宅の日記 - 続・「サニタイズ言うなキャンペーン」とは

    SQLインジェクションの例

    入力されたIDを元に検索を行う場合、下記の様のコードを書いた場合を想定します。

    1
2
3
4
    // 入力された ID で検索を行うクエリ

const sql = "select * from user where id = " + id;
const result = oracle.exec(sql);

    idに、"1 or 1 = 1" という文字列が入力された場合、下記のようなクエリが実行されます。

    1
    select * from user where id = 1 or 1 = 1;

    どうでしょうか？このクエリは、idが1であるユーザーを検索するのではなく、userテーブルの全てのレコードを返すクエリになってしまいます。

    これならまだましで、次のような値が入れられると、

    id: "1; delete from user;"

    テーブルからデータを全部削除してしまう SQL になります。

    1
    select * from user where id = 1; delete from user;

    これは、SQLが;でクエリが終了し、その後もクエリとして連続実行されるためです。

    そもそも、入力値が"1 or 1 = 1"なのであれば、その文字列で検索するのが正しいので、文字をエスケープしていないのはバグです。セキュリティ以前の問題です。

    対策

    外部から入ってきて、検証されていない値を、汚染されたデータと呼ぶこともあります。システムでは、この考え方が重要です。

    外部から入るデータはとは、下記があります。

    • ユーザーが入力するデータ
    • URLパス・クエリパラメーター
    • APIのリクエストパラメータ
    • データベースやファイルなどの外部ストレージから取得したデータ

    データベースのデータは、システムしか扱わないから大丈夫だろう、と思われるかもしれませんが、データベースが複数のシステムやモジュールからアクセスされることも多いです。

    そうすると、データベースに入った値が正しい保証があるとは限りません。きちんとシステムやモジュールごとにデータスキーマに基づいた前提条件について検討する必要があります。

    URLやSQL などのクエリを生成するのに文字列結合を使うのは、非常に危険です。クエリやURLを生成するライブラリを使うことをおすすめします。以下にライブラリの例を示します。

    URL Builder

    • Node.js: url (標準モジュール)
    • Java: java.net.URI (標準ライブラリ)
    • Python: urllib (標準ライブラリ)

    SQL Builder

    • Node.js: knex.js
    • Java: MyBates
    • Python: SQLAlchemy

    クロスサイトスクリプティング(XSS)、

    XSS もWebアプリケーションの攻撃手段として、メジャーなもので、特にサービスの利用者が標的になることが多いのが特徴です。

    この攻撃により、本来のサイトの中に偽のページを埋め込まれたり、Cookie の情報を盗まれたりすることがあります。

    原因

    攻撃を受ける原因としては、下記が挙げられます。

    • 外部から入力された値を、HTMLエンコードをしなかったり、JavaScriptのコードとしてそのまま出力している
    • Cookieの範囲や設定が不適切
    • セッションIDをURLに含めている

    対策

    対策としては、下記を行います。

    • 外部から入力された値をブラザ上に表示するときに、HTMLエンコードを行う。
    • <script>タグを動的に生成しない。
    • Cookie の参照範囲を検討する。
      • ドメインのみだとそれ以下のすべてのパスで参照される。パスでテナントなどを示す場合など注意。
    • Cookie のhttpOnly, secure属性を設定する。
    • HTTPヘッダで、CSP(Content Security Policy)を適切に設定する。
    • CORS を適切に設定する。XSSに対する攻撃に部分的に効果あり。
    • 外部から入るデータの検証を行う。(前述の〇〇インジェクションを参照のこと)

    CORS, CSPは、比較的容易に導入できるセキュリティ対策です。

    CORS は、外部からのアクセスを許可するドメインをホワイトリストで制限することで、ブラウザ側で意図しない外部のコンテンツのロードを防ぎます。サーバー側の保護ではないことに注意してください。

    CSPを設定することで、ブラウザ上で実行されるJavaScriptなどのソースを制限することができ、インジェクション攻撃を防ぎやすくなります。

    ただし、CSPが設定されているから、というだけでなく、適切な設定がサれていないと意味ないので、それぞれの設定値の意味を理解して設定する必要があります。

    これらを手軽に設定するツールや、フレームワークの設定があります。Node.jsのものですが、紹介しておきます。

    helmet - npm

    クロスサイトリクエストフォージェリ(CSRF)

    XSS と似ていますが、こちらはサイト自体への攻撃ではなく、攻撃側が用意したWebページやURLから、ターゲットのサイトに対してリクエストを送信することで、攻撃を行います。

    被害者が攻撃者が用意したリンクやページなどにアクセスすることで、ターゲットのサイトに対して、正しい認証で攻撃者が用意したアクションが実行されてしまう、というのが特徴です。

    3分でわかるXSSとCSRFの違い - Qiita

    被害の例としては、下記が挙げられます。

    • パスワードを攻撃者が設定したものに変更される
    • 攻撃者の用意したコメントが、被害者のアカウントで掲示板に掲載される。
    • ショッピングサイトなどで、勝手に商品が購入される。

    原因

    ユーザーが、ターゲットのサイト（A）にログインすると、その認証情報がCookieに保存されます。

    その後の (A) へのリクエストは、Cookieの認証情報が一緒に送信されるため、A 側では、そのリクエストが認証済みのものであると判断します。

    攻撃者が用意したサイト（B）で、(A) へのリクエストを送る処理を書いて、被害者がアクセスしてそれが実行されると、 A 側で対策がされていないと、認証Cookieも正しいことから、そのリクエストを受け入れてしまいます。

    例えば、(B) が (A) へのリクエストに、パスワードを変更するリクエストを送ると、攻撃者が用意したパスワードに変更されてしまいます。

    対策

    • CORS を適切に設定する。ただし、サーバー側の保護の目的ではないことから、これだけでは不十分。
    • ページに、ワンタイムなトークンを発行し、リクエストに含め、サーバー側でそれをチェックする。
      • Webアプリフレームワークでは、この機能を提供しているものが多いです。利用するフレームワークの仕様をチェックしましょう。
    • サーバー側で、Referer で、送信元を確認する。
    • パスワードなど重要な情報の更新には、個人しか知り得ない情報（現在のパスワード）も一緒に送る仕組みとする。
      • 多くのサイトで、パスワードの変更で、現在のパスワードを入力させているのは、このためです。
    • 保険的な対策として、重要な情報の更新後に、本人にメールや通知を送る。
    • SameSite Cookie 属性を利用する。

    SameSite Cookie は、同じドメインのページからしか送られない制約を設けることができる Cookie 属性です。

    HTTP Cookie の使用 - HTTP | MDN

    意図しない公開ファイル

    Webアプリケーションでは、HTMLやCSS,JavaScript やページで表示する画像ファイルなど、公開するファイルが多くあります。

    多くのフレームワークでは、公開するファイルを特定のディレクトリに配置することで、そのディレクトリ以下のファイルを公開するようになっています。

    そこにうっかり、公開してはいけないファイルを置いてしまうと、意図しない情報が公開されてしまうことになります。

    アプリケーションで読み込む CSV ファイルや、データベースのバックアップファイルなどが、公開されてしまうことがあります。

    対策

    • 公開するディレクトリをそれとわかり易い名前にする。(public など)
    • 公開するディレクトリを書き込み禁止にする。

    セッション・データハイジャック

    セッション・データハイジャックは、セッションIDを盗まれることで、攻撃者が被害者のアカウントとしてログインできるようになる攻撃です。

    XSS で盗まれるパターンもありますが、意外と多いのが推測されることです。

    セッションIDにの生成ルールで、日付や連番などを元にしていると、推測されやすくなります。

    同じように、データのIDに連番を採用していると、データの存在を推測されやすくなります。

    対策

    対策として、ランダムな文字列を生成するなど、推測されにくいようにすることが重要です。

    連番はこの脆弱性の他に、そのチェックや並列で非同期な処理を行うようなモジュールがあると、実装が複雑になり、バグの原因にもなります。

    UUID v4 やそれに類する生成アルゴリズムや、そのツールを使うことをおすすめします。

    UUID - Wikipedia

    おすすめ Node.jsパッケージ

    uuid - npm

    nanoid - npm

    不要な情報の公開と送信

    Webアプリケーションでは、ページの情報をHTMLなどで表示しますが、表示する以外の隠す情報もあるでしょう。

    しかし、其の隠したつもりの情報も、ブラウザの開発者ツールなどを使うと簡単に判明してしまします。

    ここに、パスワードやユーザーに秘匿な情報などが含まれる場合、情報漏洩につながります。

    また、送信する情報についても、不要な情報が含まれ、それが送信されることで、意図しないデータの更新につながることもあります。

    特に、UI上でユーザー変更できない値でも、それをサーバーに送って、その値を信用して処理を行う場合など、攻撃の対象になります。

    いくら HTML 上で編集不可にしても、開発者ツールなどでHTMLを書き換えられますし、そんなことしなくても、リクエストのデータを任意の値に変えることは容易だからです。

    • APIの脆弱性はどの程度危険なのか、どうすれば攻撃を防げるのか：API実装の落とし穴に要注意 - ＠IT
    • アプリケーション開発で「APIセキュリティ」を考慮すべき理由：セキュリティインシデントの原因となる「安全ではないAPI」をどう減らすか - ＠IT

    対策

    • ページヘは必要最低限の値を返すようにする。
    • リクエストには、必要最低限の値を含め、それ以外の値は無視したりエラーとするようにする。
    • APIの設計をユースケースに合わせたものにする。
    • API設計でバリデーション、型を定義し、APIベースでのテストを行う。

    それぞれ、そもそもがAPIが適切に設定されていない可能性が大きいです。

    例えば、あるデータのステータスを書き換えるだけのアクションなのに、データの更新処理のAPIが値でも書き換えられるAPIしかなく、それを使っている場合、 そのAPIを使うと、意図しないデータの更新が行われてしまいます。

    APIを設計するときに、単純なデータのCRUDのものしか用意していないと、こういった問題が起きやすくなります。また、ケースごとのチェックの分岐で対処できなくはないですが、 アクションがフエルと非常に複雑な実装になります。

    ユースケースやアクションごとのAPIを設計することで、この問題を回避できます。

    さらにそうすることで、APIの設計が必要で、APIベースでのテストに意識がむけられるようになります。

    システムの入り口は、WebUIではなく、APIである、という認識が必要です。

    WebUI を操作してテストを行うのは、UTでもITでもなく、EtoEテスト（総合テスト）の段階です。

    おしゃべりなエラーメッセージ

    何かしらのエラーが発生した場合、スタックトレースなど詳細までクライアントに返してしまうと、攻撃者にとっては、攻撃の手がかりになります。

    例えば、SQLのエラーが発生した場合、SQLの文がそのまま返ってきてしまうと、攻撃者はSQLインジェクションの攻撃に使えます。

    対策

    • エラーメッセージは、最低限でユーザーにわかりやすいものにする。
    • エラーログをサーバー側で取得し、追跡できる仕組みを作る。
    • デバッグモードと、本番モードでログの出力方法を切り替えられるようにする。

    古いOS、ミドルウェア、パッケージの既知の脆弱性をついた攻撃

    古いOSやミドルウェア、パッケージを使用していると、そのバージョンに対する脆弱性が発見されている可能性があります。

    その脆弱性を突かれて、システムが乗っ取られたり、データが盗まれたりすることがあります。

    定期的なバージョンの確認と、更新作業が必要ですが、思っているだけでできていないことも多いです。

    また、クラウドサービスを使用している場合も注意です。例えば lambda は、様々な言語をサポートしていますが、サポートしているバージョンの下限が変わることがあります。

    そんなときに慌てないように、運用設計・計画時に、これらのバージョンの確認やバージョンアップを含むようにします。

    まとめ

    システムのセキュリティとは広範囲に要素があり、これができていたらOKというものではありません。

    仕様設計、アーキテクチャ、システム設計、実装、デプロイメントそれぞれで異なるセキュリティ対策が必要です。

    まずは脆弱性を知り、それに対する対策をしっかり行っていくことが大切になります。

    Webアプリとして、上でも述べました代表的な対策を挙げておきます。

    • HTTPヘッダ CORS, CSP の適切な設定
    • Cookie のドメイン範囲、http-only secure属性の設定
    • 認証セッションの取扱いの設計
    • バリデーションとエンコーディング
    • 必要最低限のAPI設計とAPIベースでのテスト
    • OS,ミドルウェア等の定期バージョンアップ計画
